# Методист РУП — проверка учебного плана (дизайн)

**Дата:** 2026-06-01
**Статус:** утверждён к реализации

## 1. Цель

Добавить в проект «ИИ-наблюдатель урока» отдельный раздел **«Методист РУП»** —
проверку текстовых учебно-методических документов (ОӘЖ / КТЖ / программы практики)
через Claude по стандартам ТжКБ РК. Прообраз — самостоятельная страница
`oazh-checker.html`, которую переписываем на архитектуру проекта (ключ Anthropic на
сервере, строгий JSON, prompt caching), без браузерного ключа и CDN/Babel.

Документ-проверка по природе быстрая и одношаговая (один вызов Claude по тексту),
поэтому обработка **синхронная**, без фоновой очереди и поллинга (в отличие от
анализа аудио-урока).

## 2. Принятые решения

| Вопрос | Решение |
|---|---|
| Интеграция | Отдельный раздел/страница в том же фронте и бэкенде |
| Ввод | Вставка текста **или** загрузка файла (.docx/.pdf/.xlsx/.txt) |
| Язык | Двуязычный (қаз/рус), переключатель как в анализе урока |
| Хранение | С историей в БД (PostgreSQL), как уроки |
| Обработка | Синхронный эндпоинт (без фоновой задачи/поллинга) |

## 3. Бэкенд

### 3.1 Новые/изменённые модули (`backend/app/`)

- **`llm_utils.py`** (новый) — общий хелпер `extract_json(text)` вынесен из
  `analysis.py` (там сейчас `_extract_json`); переиспользуется анализом урока и
  проверкой плана. `analysis.py` импортирует его вместо локальной копии.
- **`doc_extract.py`** (новый) — извлечение текста из загруженного файла:
  - `.docx` → `python-docx` (абзацы + текст таблиц);
  - `.pdf` → `pypdf` (текст страниц);
  - `.xlsx` → `openpyxl` (склейка непустых ячеек по строкам — РУП часто в Excel);
  - `.txt` / прямой текст → как есть.
  - Неподдерживаемый формат или ошибка парсинга → `ValueError` с понятным текстом.
- **`plan_check.py`** (новый) — двуязычный системный промпт «Бас Әдіскер /
  Главный методист» (3 критерия), вызов Claude с `cache_control`, парсинг через
  `extract_json`, Pydantic-валидация. Возвращает `(PlanCheckResult, raw_text, is_raw)`.
- **`models.py`** — новая таблица `PlanCheck`.
- **`schemas.py`** — новые Pydantic-схемы.
- **`main.py`** — три эндпоинта.

### 3.2 Модель `PlanCheck`

| Поле | Тип | Назначение |
|---|---|---|
| `id` | int PK | идентификатор |
| `title` | str | название документа |
| `language` | str | `ru` \| `kk` |
| `source_filename` | str \| null | имя загруженного файла (если был) |
| `input_text` | text | извлечённый/вставленный текст плана |
| `status` | str | `done` \| `error` |
| `verdict` | str \| null | нормализованная резолюция |
| `summary` | text \| null | краткое заключение |
| `errors` | JSON | список `{type, description, example}` |
| `optimized_plan` | text \| null | эталонная версия |
| `raw_response` | text \| null | сырой ответ модели |
| `is_raw` | bool | модель вернула не-JSON (фолбэк) |
| `error_message` | str \| null | текст ошибки при `status="error"` |
| `created_at` | datetime | время проверки |

Схема создаётся автоматически при старте, как и для `Lesson`.

### 3.3 Pydantic-схемы (`schemas.py`)

- `PlanError { type: str, description: str, example: str | None }`
- `LLMPlanCheck { verdict, summary, errors: list[PlanError], optimized_plan }` —
  валидация ответа Claude.
- `PlanCheckOut` — отдача результата на фронт (поля модели + список ошибок).
- `PlanCheckListItem` — короткая запись для истории (`id, title, verdict, status, created_at`).

### 3.4 Промпт (`plan_check.py`)

Неизменная часть (system, `cache_control: ephemeral`): персона «Бас Әдіскер /
Главный методист», три критерия — **педагогическая логика**, **дубликаты**
(СӨЖ/СПОӨЖ), **глаголы Блума** (активные vs пассивные формулировки) — и строгая
JSON-схема ответа (`verdict`, `summary`, `errors[]`, `optimized_plan`).

Язык ответа задаётся в user-сообщении в зависимости от `language`
(«Отвечай на русском.» / «Қазақ тілінде жауап бер.»).

`verdict` нормализуется к трём состояниям (по образцу `resolveVerdict`):
`ҚАЙТА_ӨҢДЕУГЕ` → red, `ІШІНАРА_ҚАБЫЛДАНСЫН` → amber, `ТОЛЫҚ_БЕКІТІЛСІН` → green.

### 3.5 Эндпоинты (`main.py`)

| Метод | Путь | Назначение |
|---|---|---|
| `POST` | `/plans/check` | multipart: `title`, `language` (`ru`\|`kk`), `text?`, `file?` → результат проверки |
| `GET` | `/plans` | список проверок (история) |
| `GET` | `/plans/{id}` | одна проверка с полным результатом |

Поток `POST /plans/check`:
1. Валидация: задан `text` **или** `file`, иначе 400.
2. Если `file` — извлечь текст через `doc_extract` (ошибка → 400).
3. Вызвать `plan_check` (Claude → JSON → валидация).
4. Сохранить `PlanCheck` (`status="done"`, либо `is_raw` при не-JSON).
5. Вернуть `PlanCheckOut`.

## 5. Обработка ошибок

- Пустой ввод (ни текста, ни файла) → 400 с сообщением.
- Неподдерживаемый формат / битый файл → 400, текст в баннере.
- `ANTHROPIC_API_KEY` не задан → 500 с явным сообщением.
- Claude вернул не-JSON → фолбэк `is_raw`: показать сырой ответ в карточке
  «Ответ», запись `status="done"`.
- Прочее исключение → запись `status="error"` + `error_message`, фронт
  показывает карточку ошибки.

## 7. Тесты (`backend/app/tests/`)

- `doc_extract` — по мини-файлу каждого формата (docx/pdf/xlsx) → ожидаемый текст.
- `plan_check` — парсинг/валидация при замоканном Claude: корректный JSON, JSON в
  ```-обёртке, мусор → `is_raw`-фолбэк.
- `POST /plans/check` — с текстом и с файлом (Claude замокан): сохранение в БД и
  форма ответа.

## 8. Вне области (YAGNI)

- Асинхронная очередь/поллинг для проверки плана.
- Браузерный ввод API-ключа (ключ только на сервере).
- Экспорт результата в PDF/Word, диффы версий плана, авторизация пользователей.
