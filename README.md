# 🎧 ИИ-наблюдатель урока

MVP веб-приложения для анализа качества преподавания по аудиозаписи занятия.
Преподаватель или завуч загружает запись урока (`.mp3` / `.wav`), а система:

1. **транскрибирует** речь (faster-whisper) с **диаризацией** — разделением говорящих (pyannote.audio);
2. **анализирует** транскрипт через **Claude** (Anthropic API);
3. показывает **дашборд** с метриками и рекомендациями.

## 📊 Что измеряется

| Метрика | Описание |
|---|---|
| **Баланс речи** | % времени говорит преподаватель vs студенты |
| **Вовлечённость** | количество вопросов, число активных студентов |
| **Типы вопросов** | закрытые (да/нет) vs открытые/проблемные |
| **Покрытие плана** | какие ключевые понятия из заданного списка прозвучали |
| **Структура урока** | наличие вступления, объяснения, закрепления, итога |
| **Рекомендации** | конкретные советы преподавателю (на русском) |

## 🏗 Архитектура

```
            ┌─────────────┐      multipart       ┌──────────────────────────┐
  Браузер ──┤  Frontend   ├──── POST /upload ────┤        Backend           │
 (React +   │  (Vite,     │◄─── GET /lessons ────┤       (FastAPI)          │
  recharts) │  recharts)  │     (поллинг)        │                          │
            └─────────────┘                      │  BackgroundTask:         │
                                                 │   1. faster-whisper (STT)│
                                                 │   2. pyannote (диаризация)│
                                                 │   3. Claude (анализ→JSON) │
                                                 └───────────┬──────────────┘
                                                             │ SQLAlchemy
                                                       ┌─────▼─────┐
                                                       │ PostgreSQL│
                                                       └───────────┘
```

**Поток обработки асинхронный.** `POST /lessons/upload` сохраняет файл, создаёт
запись со статусом `pending` и сразу возвращает `lesson_id`. Тяжёлая обработка
(STT → диаризация → LLM) идёт в фоне; статус меняется
`pending → processing → done | error`. Фронтенд опрашивает `GET /lessons/{id}`
раз в 3 секунды, пока урок не завершится.

### Ключевые технические решения

- **Длинные записи** обрабатываются без ручного чанкинга: faster-whisper с
  VAD-фильтром сам режет аудио по паузам, поэтому память не растёт линейно с
  длительностью. Транскрипт целиком влезает в окно Claude (200k токенов).
- **Сопоставление текста и говорящих**: Whisper и pyannote работают независимо,
  затем каждый текстовый сегмент привязывается к спикеру по максимальному
  перекрытию таймкодов (`transcription.py`).
- **Роли**: самый «говорливый» спикер помечается как `ПРЕПОДАВАТЕЛЬ`, остальные —
  `СТУДЕНТ N` (эвристика, достаточная для MVP).
- **Строгий JSON от LLM**: системный промпт фиксирует схему ответа; парсер
  устойчив к markdown-обёрткам, результат валидируется Pydantic-схемой перед
  записью в БД (`analysis.py`).
- **Prompt caching**: неизменная инструкция помечена `cache_control` — повторные
  запросы дешевле и быстрее.
- **Деградация без диаризации**: если `HF_TOKEN` не задан, приложение всё равно
  работает — просто без разделения говорящих.

## 🚀 Запуск через docker-compose

### 1. Подготовка

```bash
cp .env.example .env
```

Откройте `.env` и впишите как минимум `ANTHROPIC_API_KEY`.

Для **диаризации** (разделения говорящих) нужен `HF_TOKEN`:
1. создайте токен на https://huggingface.co/settings/tokens;
2. примите условия моделей (один раз, кнопка «Agree»):
   - https://huggingface.co/pyannote/speaker-diarization-3.1
   - https://huggingface.co/pyannote/segmentation-3.0
3. впишите токен в `HF_TOKEN` в `.env`.

> Без `HF_TOKEN` приложение запустится, но все реплики будут отнесены к одному
> говорящему, и баланс речи оценивается приблизительно.

### 2. Сборка и старт

```bash
docker compose up --build
```

- Frontend: <http://localhost:5173>
- Backend (Swagger UI): <http://localhost:8000/docs>

> ⏳ При первом запуске backend скачивает модель Whisper (`medium`, ~1.5 ГБ) и
> модели pyannote. Они кэшируются в volume `backend_data`, поэтому следующий
> старт быстрый.

### 3. Использование

1. В левой панели заполните **тему урока**, выберите **язык** (русский/қазақша),
   введите **ключевые понятия** плана (по одному на строку) и выберите аудио- или
   видеофайл (аудио: `.mp3`, `.wav`, `.m4a`, `.ogg`, `.flac`; видео: `.mp4`,
   `.mov`, `.mkv`, `.webm` — из видео извлекается только звуковая дорожка;
   лимит размера файла: `MAX_UPLOAD_MB`, по умолчанию 2048 МБ).
2. Нажмите **«Загрузить и проанализировать»**.
3. Урок появится в истории со статусом обработки; по готовности откроется
   дашборд с графиками и рекомендациями.

## 🔌 API

| Метод | Путь | Назначение |
|---|---|---|
| `POST` | `/lessons/upload` | аудио + тема + ключевые понятия → `{lesson_id, status}` |
| `GET` | `/lessons/{id}` | статус и результат анализа урока |
| `GET` | `/lessons` | список всех уроков |
| `GET` | `/health` | проверка живости |

Поля формы `POST /lessons/upload` (`multipart/form-data`):
`title`, `language` (`ru`\|`kk`), `key_concepts` (текст: понятия через перенос
строки/запятую, либо JSON-массив), `file`.

## ⚙️ Конфигурация (переменные окружения)

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `ANTHROPIC_API_KEY` | — | **обязательно**, ключ Anthropic API |
| `ANTHROPIC_MODEL` | `claude-sonnet-4-6` | модель Claude |
| `HF_TOKEN` | — | токен HuggingFace для диаризации |
| `WHISPER_MODEL_SIZE` | `medium` | размер модели Whisper (`tiny`…`large-v3`) |
| `WHISPER_DEVICE` | `cpu` | `cpu` или `cuda` |
| `WHISPER_COMPUTE_TYPE` | `int8` | тип вычислений (`int8`, `float16`…) |
| `VITE_API_URL` | `http://localhost:8000` | URL backend для браузера |
| `CORS_ORIGINS` | `*` | разрешённые origin'ы (через запятую) |
| `MAX_UPLOAD_MB` | `2048` | мягкий лимит размера загружаемого файла, МБ |

## 🧭 Заметки и ограничения MVP

- **Производительность**: на CPU `medium` обрабатывает запись примерно за
  0.3–1× её длительности. Для ускорения используйте GPU (`WHISPER_DEVICE=cuda`,
  `WHISPER_COMPUTE_TYPE=float16`) или меньшую модель.
- **Фоновая очередь** реализована через FastAPI `BackgroundTasks` (в одном
  процессе). Для продакшена вынесите её в Celery/RQ с отдельными воркерами —
  тогда обработка переживёт перезапуск API и масштабируется.
- **Схема БД** создаётся автоматически при старте. Для эволюции схемы в проде
  подключите Alembic.
- **Точность ролей** зависит от диаризации; для спорных записей стоит дать
  пользователю возможность поправить разметку говорящих вручную.
